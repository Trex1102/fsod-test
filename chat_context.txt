16) PCB vs online prototype/memory adaptation clarification
- You asked whether “prototype/memory adaptation” is already covered by DeFRCN PCB.
- Clarified:
  - PCB is prototype-based score calibration, but it is mostly offline/fixed at test time in this repo.
  - Prototypes are built once from support data; they are not updated online from test stream.
  - PCB adjusts the score of the already predicted class using cosine similarity to that class prototype.
  - It does not perform full nearest-prototype reclassification across all classes, and does not update detector weights online.

17) Self-supervised TTT with few test inputs concern
- You asked whether self-supervised TTT is unreliable because test-time input count may be too small.
- Clarified:
  - Yes, this is a core challenge; with very few test samples, updates are noisy and can overfit/drift.
  - TTT is more stable when there is a stream/batch of unlabeled target-domain images.
  - In truly low-input settings, lightweight adaptation (e.g., BN/statistics, tiny LR, few steps, limited parameter subsets, confidence gating) is usually safer than full model updates.

18) Pseudo-label self-training: "how?" clarification
- You asked how pseudo-label self-training is done in practice for detection.
- Discussed practical loop:
  - run current detector on unlabeled target images,
  - filter predictions by confidence / NMS / class-specific thresholds,
  - convert kept predictions to pseudo GT (hard labels or soft targets),
  - fine-tune selected model parts with supervised-style losses (optionally weighted by confidence),
  - refresh pseudo-labels and repeat for a few rounds.
- Key stabilization ideas noted:
  - teacher-student (EMA teacher),
  - high-precision threshold first then relax,
  - weak/strong augmentation consistency,
  - limit updates to safer subsets (e.g., heads/BN/affine) to reduce drift.

28) Terminology clarification: “true” vs “not-true” test-time adaptation
- You asked what “true test-time adaptation” means.
- Clarified:
  - “True” test-time adaptation usually means model parameters (or at least adaptive states like BN affine/stats) are updated online using unlabeled test inputs.
  - Methods that only change inference behavior without parameter/state updates (e.g., test-time augmentation averaging, score calibration, threshold/NMS tuning) are not adaptation in the strict sense.
  - “False test-time adaptation” is not a formal term; better wording is “adaptation-free test-time inference” or “test-time tricks without learning.”

- Adaptation-free point:
  - adaptation-free/training-free FSOD methods do exist and can be very competitive (including recent foundation-model based approaches),

36) Why strict FSOD test-time learning methods are rare
- You asked why there are not many methods beyond PS-TTL.
- Discussed main reasons:
  - FSOD itself is a niche subfield; combining FSOD + test-time adaptation narrows the audience further.
  - high instability/noise: pseudo-label errors in detection (box + class) can quickly collapse adaptation with few shots.
  - protocol ambiguity and fairness concerns: adaptation can blur boundaries between test-time inference and extra training/data leakage.
  - expensive evaluation: multi-shot/multi-seed FSOD benchmarking plus online adaptation is costly and hard to reproduce.
  - engineering complexity: two-stage detectors need careful adaptation control (RPN/ROI/backbone) to avoid catastrophic drift.

37) Clarification requested: leakage vs valid test-time adaptation
- You asked for a clearer explanation of how test-time adaptation can be confused with extra training.
- Clarified core distinction:
  - valid TTA: only unlabeled test inputs are used at inference/adaptation time under a clearly stated protocol.
  - leakage/extra training: any direct or indirect use of target labels/metrics/split knowledge to tune adaptation decisions.
- Typical leakage examples discussed:
  - tuning thresholds/steps/early stopping on target test AP,
  - choosing best checkpoint on query/test labels,
  - using future samples in an online claim,
  - using test annotations to build/support prototypes,
  - extensive target-specific hyperparameter search without a separate unlabeled validation protocol.

52) Opinion on OCBE (RD++) style multiscale fusion for DeFRCN/FSOD
- You shared OCBE/BN_layer architecture from RD++ anomaly detection and asked whether it is suitable for multiscale fusion.
- Assessment recorded:
  - conceptually useful idea: multi-scale alignment + learned fusion into a unified latent can improve representation quality.
  - direct transfer risk for FSOD detection: OCBE aggressively compresses spatial resolution (to 8x8), which is likely harmful for RPN proposal recall and small-object localization.
  - full OCBE block is relatively heavy (concat 3072 channels + deep bottlenecks), increasing overfitting risk in few-shot detector fine-tuning.

84) Reviewed `vae-fsod.pdf` (Generating Features with Increased Crop-related Diversity for FSOD)
- Core claim:
  - FSOD classifiers (incl. DeFRCN) lack robustness to crop-related variation (proposal quality/coverage variability), especially in low-shot regimes.
- Method:
  - Trains a class-conditional VAE on RoI features from base classes.
  - Adds latent rescaling: z_tilde = (z / ||z||) * g(IoU), where IoU is proposal-vs-GT overlap proxy for crop difficulty.
  - Uses semantic class embeddings (CLIP in main results) to condition generation.
  - Generates synthetic novel-class features by varying latent norm (difficulty levels), then fine-tunes DeFRCN classifier with added generated-feature classification loss.
- Experimental setup:
  - Faster R-CNN / ResNet-101 features, RoI feature dim 2048.
  - Latent dim 512; semantic embedding dim 512.
  - Generates k=30 features per class.
- Key results against DeFRCN baseline:
  - VOC nAP50 (split1): 1-shot 56.3 -> 62.1 (+5.8), 2-shot 60.3 -> 64.9 (+4.6), 3-shot 62.0 -> 67.8 (+5.8).
  - COCO nAP: 1-shot 6.6 -> 9.5 (+2.9), 2-shot 11.7 -> 13.7 (+2.0); gains taper at higher shots.
  - Hard-case AP50~75 (VOC split1): 1-shot 16.6 -> 18.8, 2-shot 13.3 -> 16.4, 3-shot 15.2 -> 19.2.
- Interpretation for vanilla DeFRCN:
  - Primarily improves ROI classifier robustness/calibration to imperfect proposals.
  - Does not directly fix proposal recall bottleneck (RPN miss problem remains a separate issue).
